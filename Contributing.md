# Contributing to Tic-Tac-Toe

Thank you for your interest in contributing to **Tic-Tac-Toe**!

Contributions of all kinds are welcome. Whether you're fixing bugs, improving gameplay, enhancing the bot logic, optimizing the code, implementing new design patterns, or improving the documentation, every contribution helps make this project better.

If you find this project useful, consider sharing it with others, referencing it in your projects, or giving the repository a ⭐ to support its development.

Please remember to follow our **[Code of Conduct](https://github.com/sahil-me/Game_tic-tac-toe/blob/main/CODE_OF_CONDUCT.md)** and maintain a respectful and collaborative environment for everyone.

---

## Ways to Contribute

There are many ways you can contribute to **Tic-Tac-Toe**.

* **Feature Development** – Add new gameplay features, improve game flow, enhance the scoreboard, or introduce additional game modes.
* **Bot Improvements** – Improve bot strategies or implement smarter algorithms for different difficulty levels.
* **Bug Fixes** – Identify and resolve issues in gameplay or game logic.
* **Code Quality** – Refactor code, improve object-oriented design, or optimize existing implementations.
* **Documentation** – Improve the README, project documentation, or code comments.
* **Testing** – Add or improve unit tests to increase project reliability.

> Every contribution, no matter how small, is greatly appreciated. ❤️

---

## Reporting Bugs

If you discover a bug, please open an issue and include as much information as possible.

Helpful information includes:

* Operating System and version
* Java version
* Steps to reproduce the issue
* Expected behavior
* Actual behavior
* Screenshots (if applicable)
* Complete error messages or stack traces

Before creating a new issue, please check whether it has already been reported.

---

## Requesting a Feature

Have an idea to improve **Tic-Tac-Toe**?

Open an issue and describe:

1. The problem your feature solves.
2. Your proposed solution.
3. How the feature should work.
4. Screenshots, diagrams, or code examples (if applicable).

Well-described feature requests help us review and implement improvements more efficiently.

---

## Improving Documentation

Documentation improvements are always welcome.

Examples include:

* Fixing typos
* Improving explanations
* Updating diagrams or screenshots
* Adding setup or usage instructions
* Improving code comments

---

## Fixing Outstanding Issues

If you notice an issue with the existing code and have a fix in mind, feel free to **[start contributing](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request)** and open a Pull Request.

### Making Code Contributions

<details>

1. **Fork the Repository**
   - Go to the GitHub repository and click the **Fork** button.

2. **Clone Your Fork**

   ```bash
   git clone https://github.com/<your-username>/Game_tic-tac-toe.git
   ```

   ```bash
   cd Game_tic-tac-toe
   ```

3. **Create a New Branch**

   ```bash
   git checkout -b feature/your-feature-name
   ```

4. **Make Your Changes**

   - Write clean, readable, and maintainable code.
   - Follow the existing project structure and coding style.
   - Preserve the project's object-oriented design and design patterns.
   - Test your changes before committing.

5. **Commit Your Changes**

   ```bash
   git add .
   git commit -m "Describe your changes"
   ```

6. **Push Your Branch**

   ```bash
   git push origin feature/your-feature-name
   ```

7. **Create a Pull Request**

   Submit a Pull Request with a clear description of:

   - What you changed
   - Why the change was made
   - Screenshots or terminal output (if applicable)

8. **Respond to Feedback**

   - If reviewers suggest improvements, update your Pull Request accordingly.

</details>

---

## Development Guidelines

When contributing, please try to follow these principles:

- Keep classes focused on a single responsibility.
- Prefer extending existing abstractions over duplicating code.
- Follow object-oriented design principles where appropriate.
- Maintain consistency with the project's existing design patterns.
- Ensure new functionality does not break existing gameplay.

---

## Becoming a Maintainer

This project welcomes contributors who are interested in helping maintain and improve the application.

Contributors who consistently provide high-quality improvements, review pull requests, improve documentation, improve test coverage, and actively participate in the project may be invited to become maintainers.

We appreciate every contribution and look forward to collaborating with you! 😊
